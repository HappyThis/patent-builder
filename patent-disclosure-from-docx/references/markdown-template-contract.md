# Markdown 模板约定

本文档定义本 skill 使用的 Markdown 模板约定。目标不是做模板解析，而是提供一份对 LLM 友好、同时适合通用专利交底写作的结构约束。

## 模板目标
- 模板应服务于技术交底写作，而不是排版渲染。
- 模板应尽量让章节意图清晰、顺序稳定、提示语明确。
- 模板应便于 LLM 直接阅读和按章节补全。
- 模板应帮助模型把专业术语讲准确，同时在必要时补足概念解释，方便专利代理人理解。
- 模板不应默认场景一定是计算机、软件或算法方案，应同时适用于结构、装置、工艺、材料、控制和方法类交底。
- 模板应优先保证通用性和必要性，不为了“看起来完整”引入并非每个交底书都需要的固定章节。

## 推荐结构
- 使用 `#` 作为文档标题。
- 使用 `##` 作为一级章节。
- 默认不强制使用 `###`；只有确实有助于说明方案时再拆分。
- 每个章节下面可以包含一句到两句提示语，说明该节需要写什么。
- 在确有帮助时，可以在章节内部插入简短公式、伪代码块或流程化步骤列表，用于增强逻辑表达。

## 推荐章节
模板可根据场景裁剪，但通常建议至少包含以下内容：
- 技术领域与背景
- 要解决的技术问题
- 技术方案
- 关键创新点与保护点
- 技术效果与必要说明

## 提示语写法
- 推荐使用自然语言提示语，例如 `[说明本方案的整体流程]`。
- 提示语应聚焦写作意图，不要使用过于抽象的变量名。
- 提示语可以保留在模板中，但在最终输出时应尽量被真实内容替换。
- 提示语可以直接提醒模型解释关键概念，例如“首次出现的术语请补一句定义或职责”。

## 语气与术语要求
- 文风应像专业技术人员向专利代理人做交底，而不是像市场文案或学术论文摘要。
- 保留必要专业术语，但不要默认读者知道所有缩写、结构名称、材料名称、算法名、协议名或内部业务概念。
- 对首次出现且理解门槛较高的概念，建议补一句“是什么”或“在本方案中起什么作用”。
- 同一对象尽量使用统一名称，避免频繁切换近义词。
- 如果使用公式、伪代码或流程化表达，必须让读者看得出它们分别对应哪一段机制、哪一步处理或哪组条件。

## 用户自定义模板约束
- 优先支持 Markdown 模板。
- 章节命名可以自定义。
- 章节顺序可以自定义。
- 如果模板缺少必要章节，生成时允许补充，但应尽量不破坏原始结构。

## 不推荐的写法
- 不推荐把模板写成复杂变量表，例如十几行 `{{field_name}}`。
- 不推荐把重要说明都堆在文档最底部，而正文没有章节提示。
- 不推荐混用过多层级的标题，避免结构过深影响理解。
- 不推荐只写“提升效率”“提高精度”“增强强度”这类结果，而不解释为什么会产生这些效果。

## 输出要求
- 最终输出应是一份完整 Markdown 文档。
- 不应残留明显的“待填写”“TODO”“请补充”等占位提示。
- 若某个补充章节确实无内容，可在成稿中删除，而不是保留空标题。
- 成稿应足以支持专利代理人继续撰写，而不是只给出一组零散技术点。
- 公式、伪代码和流程化表达应作为辅助说明手段，而不是替代正文的主要表达方式。
