# 专利交底书生成（纯 Markdown 模式）

## 概述
通过对话先收敛技术方案，再按 Markdown 模板直接输出完整专利交底书。  
全部协作状态保留在模型上下文中，直接形成最终成稿。  
写作目标不是堆砌技术名词，而是把方案讲清楚，让专利代理人既能把握专业要点，也能继续展开专利撰写。  
面向用户的最终交付物：仅 `markdown`。

## 模板来源
- 默认情况下，使用内置模板 [default-disclosure-template.md](references/default-disclosure-template.md)。
- 如果用户提供自定义 Markdown 模板，则优先使用用户模板。
- 若用户模板结构不完整，优先保留其章节命名和顺序，再以最少章节补足必要技术交底内容。

## 输入
- 用户提供的交底内容想法。
- 可选约束条件（截止时间、命名、保密要求、范围等）。
- 可选用户自定义 Markdown 模板。

## 工作流

### 第 1 步：确认模板来源
- 先判断用户是否提供了自定义 Markdown 模板。
- 若未提供，则直接采用内置默认模板。

### 第 2 步：先收敛写作方向（在上下文中完成）
- 先询问用户对交底内容的初始想法。
- 推荐 2 到 4 个写作方向，用户可以选择，也可以自定义方向。
- 在方向层面先讲清核心问题、总体思路、适用对象和潜在保护点，不要过早进入成稿措辞。
- 对于关键专业概念、缩写、部件、结构、材料、工艺步骤、算法、模型或业务对象，要同步澄清“它是什么”和“它在本方案里负责什么”。
- 用户确认方向后，进入写作方案阶段，而不是直接进入成稿。

### 第 3 步：先给出写作方案并让用户确认
- 基于用户选定方向，先输出一版“写作方案”。
- 写作方案本质上是交底书中“技术方案”部分的可实施草案，不是文案提纲。
- 写作方案至少要包含：
  - 方案目标与核心对象
  - 输入是什么、如何获得
  - 输出是什么、用于什么
  - 核心结构、步骤、工艺或控制逻辑
  - 关键参数、变量、条件或约束
  - 每个关键参数或变量的含义、来源和确定方式
  - 当前仍需用户确认、补充或修正的点
- 写作方案给出后，必须让用户评估、纠正和补充；用户未确认前，不进入最终写作。

### 第 4 步：检查写作方案的合理性与可实施性
- 写作方案必须先通过合理性检查，再进入成稿。
- 检查重点包括：
  - 关键对象是否都已定义
  - 关键步骤是否写清输入、动作、输出
  - 关键结构或部件是否写清作用关系
  - 关键参数、阈值、系数、权重、比例、时间、温度、压力、浓度、速度等是否说明来源与确定方式
  - 公式中的每个符号是否说明“是什么、怎么来、拿来做什么”
  - 若使用伪代码、流程化步骤或公式，是否确实帮助说明逻辑，而不是只增加形式感
  - 判断条件、触发条件、边界条件和异常处理是否交代清楚
  - 技术效果是否能回溯到前文机制
- 如果存在“看似合理但无法落地”的情况，应继续追问或显式列出缺口，而不是直接写成交底书。

### 第 5 步：在上下文中生成最终 Markdown
在用户明确批准写作方案后：
- 直接生成完整 Markdown 成稿。
- 优先沿用模板中的章节标题、章节顺序与写作提示。
- 默认模板只保留最必要部分，重点是把问题、方案、关键点和效果讲清楚，而不是追求章节齐全。
- 如果模板中存在提示性占位语，应在最终输出时将其替换为真实内容，不保留“待填写”“请说明”等引导文本。
- 当技术方案需要更强的逻辑表达时，可以适当插入公式、伪代码或流程化步骤说明，但它们必须服务于解释方案，而不是替代正文说明。
- 当使用公式、伪代码或流程化步骤时，要补充说明其适用位置、输入输出、关键变量和与正文机制的对应关系。
- 行文风格应像工程师向专利代理团队做技术交底，而不是正式专利申请文件。语气要直接、具体、面向实现。
- 术语要专业、准确、前后一致；但遇到首次出现或可能造成理解障碍的概念，应顺手补一句定义、职责或与上下游的关系。
- 对于非软件类方案，同样要写清楚结构关系、材料选择、工艺步骤、装配方式、控制条件或作用机理，避免默认模板偏向计算机场景。
- 使用“本方案……”或直接陈述句，避免“我建议……”“可选地……”“优选地……”“本发明……”或“权利要求……”这类措辞。
- 优先使用短段落、明确编号步骤、清晰的小节结构和易审阅的 Markdown 格式。

## 决策规则
- 默认使用内置 Markdown 模板；用户提供模板时优先使用用户模板。
- 全部协作信息保留在对话上下文中，最终一次性输出成稿。
- 先收集想法，再给出方向；方向确认后，必须先给出写作方案。
- 用户没有明确批准写作方案前，不进入正式写作。
- 将文档视为技术交底材料，而不是正式专利说明书。目标是把技术方案解释清楚，使专利代理人能够继续起草。
- 默认模板遵循“最小必要章节”原则，不为了完整感增加并非每个交底书都需要的固定章节。
- 在“专业性”和“可理解性”之间取中间点：保留必要专业术语，但不要默认读者知道全部上下文。
- 对首次出现的关键术语、缩写、部件、结构、材料、工艺、算法、模型、数据对象和业务概念，必要时补充简短说明。
- 不只写结论，要写清楚机制、条件、依赖关系、作用链路和关键约束。
- 禁止引入未解释来源的核心参数、公式符号、阈值、权重、系数、评分或控制量。
- 若出现参数或公式，必须尽量说明其定义、来源、确定方式和用途；暂时无法精确给出时，也要说明后续确定路径，例如试验标定、经验查表、实时测量、规则计算或模型训练。
- 若公式、伪代码或流程图式描述能够明显增强逻辑清晰度，可以使用；但不能只给表示形式，不给文字解释。
- 模板中的可见提示语、占位标签和说明性括号，在最终输出中应尽量被真实内容替换。
- 若用户模板遗漏必要章节，可在不破坏原模板主结构的前提下做最小补充。
- 对外交付物仅为最终 Markdown。

## 输出要求
- 输出必须是一份完整、可直接阅读的 Markdown 文档。
- 文档应包含明确标题和完整章节。
- 输出内容聚焦最终交底书正文，不附带规则文件、过程文件或额外报告。
- 文档应让专利代理人能够回答“方案解决什么问题、核心怎么实现、哪些点值得保护、关键术语分别指什么”这几类问题。
- 在进入最终成稿前，应先形成一版可评审的写作方案，并明确标出待确认项。
- 除非用户明确要求，否则不附带“说明”“备注”“生成过程”等额外内容。
